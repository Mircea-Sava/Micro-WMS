# Warehouse.xlsm - Complete Structure Reference

## Workbook Overview

| Sheet | Index | Purpose |
|-------|-------|---------|
| Form | 1 | User interface for entering pick/putaway orders |
| Map_Grid | 2 | Visual warehouse map (A1:AO33) with A* pathfinding overlay |
| Map_Helper | 3 | Backend database for bin locations, inventory state, and SKU registry |
| Transaction_History | 4 | Permanent log of all confirmed operations |

All VBA code lives in a single module: **Module1** (1089 lines).

---

## Sheet 1: Form

![Form sheet](images/Form.png)

**Purpose**: primary input form where users enter pick/putaway orders before confirming.

| Column | Header | Data Type | Description |
|--------|--------|-----------|-------------|
| A | SKU | Text | Item identifier (e.g., "OBJ_A", "OBJ_B") |
| B | Qty | Number | Quantity of units to pick or put away |
| C | Operation | Text | Must be "PICK" or "PUTAWAY" |
| D | Location Assigned | Text | Bin name (e.g., "A1-R1-L1-A"). Left blank for auto-assignment. Comma-separated for multi-stop. |

**UI Elements**: 3 buttons on the sheet (Confirm, Generate Map, Clear).

**Usage**: fill rows 2+ with order lines, then run `ConfirmOperations`. If Location is left blank on a PUTAWAY, the system auto-assigns a bin. If blank on a PICK, the system finds a bin containing that SKU.

## Sheet 2: Map_Grid

![Map_Grid with generated route](images/Map_Grid.png)

**Purpose**: visual 2D representation of the warehouse floor, drawn to scale from an AutoCAD drawing.

**Grid Range**: A1:AO33 (41 columns x 33 rows)

| Cell Value | Meaning |
|------------|---------|
| `1` | Wall / obstacle (impassable by A* pathfinding) |
| `2` | Dock / start point (worker entry, used as pathfinding origin) |
| Bin name (e.g., `A1-R1-L1-A`) | A storage location on the floor |
| Empty | Open floor (walkable) |

**Scale**: each cell = 0.3848m (stored as `TILE_SCALE` constant in VBA). Derived from the AutoCAD drawing ratio: 620" x 500" mapped onto 41 x 33 cells.

**Visual Elements Generated by VBA**:
- **Path lines**: colored connectors drawn by `DrawGPSPath`. Color gradient goes green to red across the route to show progress.
- **Target markers**: shapes placed at bin locations by `DrawTargetMarker`:
  - Oval = Level 1 bin
  - Triangle = Level 2 bin
  - Rectangle = other levels
  - Red fill = PICK operation
  - Green fill = PUTAWAY operation
- **SKU labels**: tiny text boxes next to markers showing the SKU name.
- **Stats box**: dark rounded rectangle in the top-right area showing "TOTAL: X.Xm" total walking distance.

**Cell AQ1**: stores the calculated total route distance (meters) after `GenerateMapPath` runs.

All generated shapes are named `PathLine_Generated` or `PathLabel_Generated` so `ClearMap` can remove them without affecting the base map.

## Sheet 3: Map_Helper

![Map_Helper sheet](images/Map_Helper.png)

**Purpose**: backend database combining the bin registry, coordinate lookup, distance/rank analysis, SKU master list, and live inventory state.

### SKU Master Registry (Columns XEU:XEV)

| Column | Header | Data Type | Description |
|--------|--------|-----------|-------------|
| XEU | SKU | Text | Registered SKU identifier |
| XEV | Capacity | Number | Maximum units this SKU can hold per bin (default 50) |

New SKUs are registered via `RegisterSKU` when first encountered during `ConfirmOperations`. The user is prompted to enter a capacity.

### Bin Database (Columns XEX:XFD)

| Column | Header | Data Type | Source | Description |
|--------|--------|-----------|--------|-------------|
| XEX | Bin Name | Text | Formula (SORT/UNIQUE) | Unique bin names extracted from Map_Grid cells |
| XEY | Grid X | Number | Formula (MAP/LAMBDA) | Column number of the bin on Map_Grid |
| XEZ | Grid Y | Number | Formula (MAP/LAMBDA) | Row number of the bin on Map_Grid |
| XFA | Distance (m) | Number | `Analyze_Map_Locations` | Round-trip A* distance from dock to bin and back |
| XFB | ABC Rank | Text (A/B/C) | `Analyze_Map_Locations` | Classification based on distance percentiles |
| XFC | Current SKU | Text | `RebuildInventoryState` | What SKU is currently stored in this bin ("EMPTY" if vacant) |
| XFD | Current Qty | Number | `RebuildInventoryState` | How many units are currently in this bin |

**Note**: Column XEW is intentionally skipped/unused.

**Dynamic Formulas** (set up by `SetupBinTable`):
- XEX2: `=SORT(UNIQUE(TOCOL(IF((A1:AO33<>1)*(A1:AO33<>2)*(A1:AO33<>""),A1:AO33,1/0),3)))` extracts all non-wall, non-dock cell values as bin names
- XEY2: `=MAP(XEX2#, LAMBDA(b, SUMPRODUCT(($A$1:$AO$33=b)*COLUMN($A$1:$AO$33))))` finds column of each bin
- XEZ2: `=MAP(XEX2#, LAMBDA(b, SUMPRODUCT(($A$1:$AO$33=b)*ROW($A$1:$AO$33))))` finds row of each bin

**ABC Ranking Thresholds** (calculated by `Analyze_Map_Locations`):
- **A** (closest): distance <= 20th percentile
- **B** (medium): distance <= 50th percentile
- **C** (farthest): distance > 50th percentile

Color coding: A = green (RGB 198,239,206), B = yellow (RGB 255,235,156), C = red (RGB 255,199,206). Applied to both the rank cells and the corresponding Map_Grid cells.

### Bin Naming Convention

Format: `{Aisle}-{Rack}-{Level}-{Side}`

Example: `A3-R2-L2-A`
- A3 = Aisle 3
- R2 = Rack 2
- L2 = Level 2
- A = Side A

![Transaction History](images/Transaction_History.png)

**Purpose**: permanent, append-only log of every confirmed operation. Used by `RebuildInventoryState` to replay and calculate current inventory.

|--------|--------|-----------|-------------|

| C | Time | Time | Transaction time (hh:mm:ss) |
| D | SKU | Text | Item identifier |
| E | Qty | Number | Quantity picked or put away |
| F | New Stock Level | Number | Snapshot of bin stock after this transaction |
| G | Type | Text | "PICK" or "PUTAWAY" |

## What the VBA Code Does

All the automation lives in a single VBA module (Module1). Here's what each part handles:

### Pathfinding

Uses the **A\* algorithm** to find the shortest path between two points on the map while avoiding walls. Give it a start and end point, and it returns the step-by-step path. See [Algorithms.md](Algorithms.md) for a full explanation.

### Bin Assignment and Inventory Helpers

These handle the logic behind finding the right bin:

- **Find a partial bin**: looks for a bin that already holds the same SKU with space left (best-fit strategy picks the fullest one)
- **Find the best empty bin**: finds the closest empty bin of a given rank (A, B, or C) from the dock
- **Find a bin for picking**: locates a bin with enough stock of the requested SKU
- **Register a new SKU**: adds a new item and its bin capacity to the system
- **Check availability**: reports how many empty bins are left per rank (A, B, C)
- **Suggest empty bin**: standalone tool that prompts for a rank and returns the closest available bin of that rank
- **Get bin distance**: looks up the pre-calculated walking distance from the dock to any bin

### Inventory State Engine

Every time an order is confirmed, the system **replays the entire Transaction_History** from top to bottom to rebuild the current inventory. PUTAWAYs add stock, PICKs subtract it. If a bin reaches 0, it's marked as EMPTY. This ensures the inventory is always accurate without manual counting.

### Confirm Operations (Main Workflow)

This is the main macro that runs when you click **Confirm**. It handles the full order process:

![New SKU registration prompt](images/New_SKU.png)

1. Registers any new SKUs (prompts you for capacity)
2. Assigns an Order ID and asks for confirmation
3. Checks that quantities don't exceed bin capacity
4. Auto-assigns bins (see Consolidation Logic below). For picks, finds a bin with enough stock.
5. Makes sure every row has a location before proceeding
6. Calculates total walking distance for the order (using A* pathfinding with route optimization)
7. Logs each line to Transaction_History (with date, time, stock levels, distances, and bin rank)
8. Rebuilds the inventory state
9. Asks if you want to generate the route map

#### Consolidation Logic (PUTAWAY Auto-Assignment)

When a PUTAWAY has no location assigned, the system tries to consolidate before opening a new bin:

**Step 1: Find a partial bin.** Scans every bin looking for one that already holds the same SKU with remaining space. If multiple bins qualify, it picks the one with the **least remaining space** (best-fit). This keeps bins as full as possible.

**Step 2: Consolidate or split.**
- If the partial bin has **enough space** for the full quantity, you're prompted: *"Consolidate to A2-R1-L1-A? (Space: 30)"*. Accept and that bin is assigned.
- If the partial bin **doesn't have enough space**, you're told to split the line: put what fits on one row and the remainder on another.

**Step 3: Assign a new bin.** If no partial bin exists (or you declined), the system asks for a preferred rank (A, B, or C) and finds the closest available empty bin. It also prevents assigning a bin that was already claimed earlier in the same order.

![Bin rank assignment prompt](images/Bin_Assign.png)

**Result**: after assignment, the bin database in Map_Helper reflects the new inventory:

![Assigned bin locations](images/Location_Assignment.png)

**Why consolidation matters**: without it, every putaway would open a new bin even if the same SKU is already sitting half-empty somewhere. This wastes bin space, spreads inventory thin, and creates extra walking during picks.

### Map Visualization

After confirming an order, you can generate a visual route on the Map_Grid sheet:

1. Reads the bins from the Form and looks up their coordinates
2. Sorts them into the best visit order using **Nearest Neighbor** (closest first)
3. Improves the order using **2-OPT** (swaps pairs of stops to reduce total distance)
4. Runs **A\*** between each stop to find wall-avoiding paths
5. Draws **colored arrows** (green = start, red = end) showing the walking route
6. Places **markers** at each bin (shape = bin level, color = PICK red / PUTAWAY green)
7. Draws the **return path** back to the dock
8. Shows a **stats box** with the total walking distance in meters

When multiple items share the same bin coordinates, markers are offset slightly so they don't overlap.

The **Clear** button removes all generated paths, markers, and form data so the next order starts fresh.

### Setup Tools (Run Once)

These are used when setting up a new warehouse layout:

- **SetupBinTable**: creates the formulas in Map_Helper that automatically extract bin names and coordinates from the map
- **Analyze_Map_Locations**: calculates the walking distance to every bin and assigns ABC ranks based on how close they are to the dock. Also color-codes the bin cells on the map (green = A, yellow = B, red = C).

## Data Flow Summary

```
User fills Form (Sheet 1)
        |
        v
ConfirmOperations (VBA)
    ├── Validates and auto-assigns bins using Map_Helper data
    ├── Logs to Transaction_History (Sheet 4)
    ├── Calls RebuildInventoryState
    |       |
    |       v
    |   Replays Transaction_History -> Updates Map_Helper XFC/XFD
    |
    └── Optionally calls GenerateMapPath
            |
            v
        Reads Form + Map_Helper coordinates
        NN + 2-OPT route optimization
        A* pathfinding on Map_Grid (Sheet 2)
        Draws path, markers, and stats on Map_Grid
```

## Setup Sequence (for a new warehouse)

1. Draw the warehouse layout in Map_Grid (A1:AO33): place "1" for walls, "2" for dock, bin names in storage cells
2. Run `SetupBinTable` to populate Map_Helper formulas
3. Run `Analyze_Map_Locations` to calculate distances and ABC ranks
4. Start entering orders in Form and running `ConfirmOperations`
